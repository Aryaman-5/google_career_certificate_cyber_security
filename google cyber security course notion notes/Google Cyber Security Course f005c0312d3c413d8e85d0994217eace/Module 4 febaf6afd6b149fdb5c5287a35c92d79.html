Databases: The Why and What

  • Data Importance: We live in a data-driven world. Databases help us store, organize, and access this data efficiently.
  • Databases vs. Spreadsheets: While spreadsheets are useful for small-scale data management, databases shine when dealing with large datasets and multiple users simultaneously.
  • Security Analyst Perspective: Databases are crucial for security analysts as they often contain information about login attempts, software, system updates, and user activity.

Relational Databases: A Closer Look

  • Definition: A relational database organizes data into tables with defined relationships between them.
  • Tables, Fields, and Records:
    • Tables: Containers for data, similar to spreadsheets.
    • Fields (Columns): Represent specific categories of information within a table (e.g., employee_id, username).
    • Records (Rows): Hold the actual data for each entry in a table.

Keys: Connecting the Dots

  • Primary Key:
    • Uniquely identifies each row in a table.
    • Must contain unique values (no duplicates).
    • Cannot be empty (no null values).
    • Example: employee_id in an employee table.
  • Foreign Key:
    • A column in one table that refers to the primary key of another table.
    • Establishes relationships between tables.
    • Can have duplicate values or be empty.
    • Example: employee_id in a 'machines' table, linking it to the 'employees' table.

SQL: The Language of Databases

  • SQL (Structured Query Language) is used to interact with and manage data within relational databases. The upcoming sections will provide hands-on experience with SQL concepts.

Introduction to databases  Jul 28, 2024

Excellent question! Understanding the components and their relationships is key to working with relational databases. Here's a breakdown:

Key Components and Relationships:

  • Tables: The foundation of a relational database. Each table stores data about a specific entity (e.g., employees, products, orders). Think of them like well-organized spreadsheets.
  • Fields (Columns): Represent the different attributes or characteristics of the entity stored in a table. For example, an 'employees' table might have fields like 'employee_id', 'name', 'department', and 'salary'.
  • Records (Rows): Each row represents a single instance of the entity defined by the table. For instance, a single row in the 'employees' table would hold all the information for one specific employee.
  • Primary Key: A special field (or combination of fields) that uniquely identifies each record in a table. It ensures that no two records are exactly alike. For example, 'employee_id' could be the primary key in the 'employees' table.
  • Foreign Key: A field in one table that refers to the primary key of another table. This establishes a link or relationship between the two tables. For instance, an 'orders' table might have a foreign key 'customer_id' that links each order to a specific customer in the 'customers' table.

How They Relate:

  • Tables are connected through relationships established using primary and foreign keys.
  • A primary key in one table is referenced by a foreign key in another table, creating a link between the data.
  • This interconnected structure allows for efficient data retrieval and analysis across multiple tables.

Example:

Imagine an online store database:

  • Customers Table: (customer_id (PK), name, email, address)
  • Orders Table: (order_id (PK), customer_id (FK), order_date, total_amount)

The 'customer_id' in the 'Orders' table is a foreign key that references the 'customer_id' (primary key) in the 'Customers' table. This connection allows you to easily retrieve all orders made by a specific customer.

SQL filtering versus Linux filtering

Previously, you explored the Linux commands that allow you to filter for specific information contained within files or directories. And, more recently, you examined how SQL helps you efficiently filter for the information you need. In this reading, you'll explore differences between the two tools as they relate to filtering. You'll also learn that one way to access SQL is through the Linux command line.

Accessing SQL

There are many interfaces for accessing SQL and many different versions of SQL. One way to access SQL is through the Linux command line.

To access SQL from Linux, you need to type in a command for the version of SQL that you want to use. For example, if you want to access SQLite, you can enter the command sqlite3 in the command line.

After this, any commands typed in the command line will be directed to SQL instead of Linux commands.

Differences between Linux and SQL filtering

Although both Linux and SQL allow you to filter through data, there are some differences that affect which one you should choose.

Purpose

Linux filters data in the context of files and directories on a computer system. It’s used for tasks like searching  for specific files, manipulating file permissions, or managing processes.

SQL is used to filter data within a database management system. It’s used for querying and manipulating data stored in tables and retrieving specific information based on defined criteria.

Syntax

Linux uses various commands and command-line options specific to each filtering tool. Syntax varies depending on the tool and purpose. Some examples of Linux commands are find, sed, cut, e grep

SQL uses the Structured Query Language (SQL), a standardized language with specific keywords and clauses for filtering data across different SQL databases. Some examples of SQL keywords and clauses are WHERE, SELECT, JOIN

Structure

SQL offers a lot more structure than Linux, which is more free-form and not as tidy.

For example, if you wanted to access a log of employee log-in attempts, SQL would have each record separated into columns. Linux would print the data as a line of text without this organization. As a result, selecting a specific column to analyze would be easier and more efficient in SQL.

In terms of structure, SQL provides results that are more easily readable and that can be adjusted more quickly than when using Linux.

Joining tables

Some security-related decisions require information from different tables. SQL allows the analyst to join multiple tables together when returning data. Linux doesn’t have that same functionality; it doesn’t allow data to be connected to other information on your computer. This is more restrictive for an analyst going through security logs.

Best uses

As a security analyst, it’s important to understand when you can use which tool. Although SQL has a more organized structure and allows you to join tables, this doesn’t mean that there aren’t situations that would require you to filter data in Linux.

A lot of data used in cybersecurity will be stored in a database format that works with SQL. However, other logs might be in a format that is not compatible with SQL. For instance, if the data is stored in a text file, you cannot search through it with SQL. In those cases, it is useful to know how to filter in Linux.

Key takeaways

Linux filtering focuses on managing files and directories on a system, while SQL filtering focuses on structured data manipulation within databases. To work with SQL, you can access it from multiple different interfaces, such as the Linux command line. Both SQL and Linux allow you to filter for specific data, but SQL offers the advantages of structuring the data and allowing you to join data from multiple tables.

SQL Queries: A Basic Example

  • Scenario: Imagine you need to find out which computer is assigned to a specific employee. You have a table named "employees" with columns like "employee_id" and "device_id."
  • Keywords:
    • SELECT: Specifies the columns you want to retrieve.
    • FROM: Indicates the table from which to fetch the data.
  • Syntax: SQL isn't case-sensitive (meaning SELECT is the same as select), but using uppercase for keywords enhances readability. Always end SQL statements with a semicolon (;).
  • Basic Query:
    SELECT employee_id, device_id
FROM employees;
  • Explanation: This query selects data from the "employees" table and returns only the "employee_id" and "device_id" columns.

"Select All" with the Asterisk (*)

  • Purpose: To retrieve all columns from a table, use the asterisk (*) as a shortcut.
  • Example:
    SELECT *
FROM employees;
  • Explanation: This query selects every column from the "employees" table.

Key Takeaways

  • You've learned how to construct basic SQL queries to retrieve specific data from a table.
  • The SELECT and FROM keywords are fundamental building blocks of SQL queries.
  • The asterisk (*) acts as a wildcard to select all columns.

Basic filters on SQL queries  Jul 28, 2024

That's a great idea to take notes! It will help you remember the important points covered. Here are some detailed notes from the lecture:

Filtering in SQL

  • What is filtering? Filtering is like selecting specific items from a collection. In SQL, it means choosing only the data that matches certain criteria.
  • Why filter? Filtering helps you focus on specific data within a larger dataset, making your queries more efficient and meaningful.
  • How to filter:
    • Use the WHERE clause after your SELECT and FROM statements.
    • Specify the filtering condition using operators.

Operators in SQL

  • What is an operator? An operator is a symbol or keyword that performs an operation on data.
  • Example: The equal to operator (=) checks if two values are equal.

Using the WHERE Clause

  • Syntax:
    SELECT column1, column2, ...
FROM table_name
WHERE condition;
  • Example: To select login attempts from the USA:
    SELECT *
FROM log_in_attempts
WHERE country = 'USA';

Pattern Matching with LIKE

  • Purpose: The LIKE operator allows you to search for patterns in text data.
  • Wildcard: The percentage sign (%) acts as a wildcard, representing any sequence of characters.
  • Example: To find offices starting with "East":
    SELECT *
FROM employees
WHERE office LIKE 'East%';

Handling Inconsistencies

  • Problem: Databases can have inconsistencies in data entry (e.g., "US" vs. "USA").
  • Solution: Use LIKE with wildcards to capture variations.
  • Example: To find login attempts from the United States, accounting for both "US" and "USA":
    SELECT *
FROM log_in_attempts
WHERE country LIKE 'US%';

Numbers, dates, and times in cybersecurity

Security analysts work with more than just string data, or data consisting of an ordered sequence of characters.

They also frequently work with numeric data, or data consisting of numbers. A few examples of numeric data that you might encounter in your work as a security analyst include:

  • the number of login attempts
  • the count of a specific type of log entry
  • the volume of data being sent from a source
  • the volume of data being sent to a destination

You'll also encounter date and time data, or data representing a date and/or time. As a first example, logs will generally timestamp every record. Other time and date data might include:

  • login dates
  • login times
  • dates for patches
  • the duration of a connection
